# C — 模型調度與動態升降級守則

> FROZEN。可用模型與塌縮規則查 ENV-FACTS §1；本檔只定義「怎麼調度」，不寫死任何模型型號以外的環境事實。
> 「子任務」定義：一個有獨立驗收條件的派工單位（一次 Agent 呼叫的任務範圍）。
> 本檔【正例／反例】以首次建置專案（單字練習平台）為藍本，屬教學示例；跨專案沿用時以規則本文為準，示例中的檔名與功能不必存在於你的專案。

## §1 指揮官不下場

主對話（指揮官）只做四件事：**決策、拆解、派工、驗收**。產生素材的勞動下放給 subagent。

判準（可機器數的）：
- 主對話**連續讀 ≥3 個原始檔**、或單檔讀超過 400 行 → 違規，改派 `scout`／Explore agent。
- 主對話對 subagent 回報以外的內容做**全 repo 掃描**（Grep 超過 2 輪找不到目標）→ 違規，改派 agent。
- 例外：讀 harness 文件與路由到的規則檔（CLAUDE.md、ENV-FACTS、SPEC 等）不受限——那是指揮官的工作素材。

【完美正例】任務「找出所有計算 XP 的地方並確認口徑一致」→ 指揮官派 scout：「列出所有 XP 計算出現的 檔案:行號 與各自公式，≤40 行回報」→ 收到 5 行結論後決策。
【典型反例】指揮官自己開 4 個 JS 檔逐行讀完貼出大段代碼再分析——context 塞進上千行原始碼，兩輪後忘記原始目標。

## §2 派工三件套（每張派工單缺一不可）

1. **明確目標與背景**：一句話目標＋為什麼做＋指名要先讀的檔案（用路徑，不貼全文；專案事實指向 ENV-FACTS §N／SPEC／OPS）。
2. **嚴格驗收條件**：可判定真假的條件列表（指令要能實跑、數字要能數）。禁止「確保品質良好」這類不可判定語句。
3. **標準回報格式**：成果路徑＋關鍵行號＋驗證指令輸出；**禁止在回報中貼大段代碼**（>15 行即違規）；不確定的事標 `UNVERIFIED`，禁止編造。

模板見 E-PROMPT-TEMPLATES.md，直接填空使用。

【完美正例】「目標：修 student.js 錯題本計數含已熟記單字的 bug（口徑見 SPEC 統計指標表）。驗收：ENV-FACTS §3 最低驗證組合全過＋用 PLAYBOOK PB-001 方式對 8 種狀態組合斷言。回報：改動檔案:行號、驗證輸出、一句話根因。」
【典型反例】「幫我看一下錯題本好像怪怪的，順便改好。」——無驗收條件，agent 會自由發揮並自稱完成。

## §3 升降級階梯

| 觸發 | 動作 |
|---|---|
| Haiku 發生 1 次工具呼叫錯誤或語法錯誤 | 同一子任務立即升 Sonnet 重派（不給 Haiku 第二次） |
| Sonnet 同一子任務連續失敗 2 次 | 帶**完整失敗軌跡**（兩次的派工單、回報、錯誤輸出）升 Opus |
| Opus 解出「會重複出現」的問題 | 把解法寫成 PLAYBOOK 條目（格式見該檔頭），之後同類任務降回 Sonnet/Haiku 批次套用，指揮官抽驗 ≥20% 樣本 |
| 同一子任務在同一層重試 | 上限 2 次（**Haiku 層除外**——Haiku 適用本表第 1 列，錯 1 次即升，不給第 2 次）；超過→依 D 矩陣 §1 判斷是「換路徑」還是「升級」，禁止第 3 次原樣重試 |

- 「失敗」定義：驗收條件任一條不過、或回報含編造（驗證指令輸出對不上）。
- 缺級塌縮：依 ENV-FACTS §1 的塌縮規則執行。
- 預設起點：機械性、已有 PLAYBOOK 配方的任務從 Haiku 起派；其餘從 Sonnet 起派；**永不**「保險起見直接用 Opus」——Opus 只能由失敗軌跡觸發。

【完美正例】Sonnet 兩次修不好時區計算 → 指揮官把兩份失敗回報原文＋SPEC 相關段落一起交給 Opus：「前兩次嘗試與失敗原因如附，請先診斷為何都失敗，再給修法。」
【典型反例】Sonnet 失敗兩次，第三次把同一張派工單原樣再丟給 Sonnet 並加一句「這次請更仔細」。

## §4 隔離驗證（實作者不得自我驗收）

- 實作 agent 回報「完成」≠ 完成。指揮官必須另派 **fresh-context 的 `verifier` agent**（.claude/agents/verifier.md）驗收，verifier 看不到實作過程、只拿到：驗收條件＋成果路徑＋ENV-FACTS §3 驗證指令。
- verifier 三板斧：**read-back**（重新打開成果檔確認內容真的存在且非截斷）→ **實跑**（跑驗收條件裡的指令、餵測資）→ **多樣本**（有數字口徑的，至少 3 個不同輸入樣本）。
- verifier 只回報 PASS/FAIL＋證據，**禁止動手修**；FAIL 由指揮官決定重派誰修。
- 例外（免派 verifier）：純文件 typo、單行註解等零邏輯改動——但仍須指揮官親自 read-back。

【完美正例】實作 agent 稱四檔語法檢查全過 → verifier 自己重跑四條 `node --check` 並貼 exit code，其中一檔其實有語法錯 → FAIL 退回。
【典型反例】實作 agent 說「我已仔細檢查，測試都通過」→ 指揮官直接採信收工（本專案根本沒有「測試」可跑，見 ENV-FACTS §3——這句話本身就是編造信號）。

## §5 平行派工

- 互不依賴的子任務同批平行派出；但兩個實作 agent 的「允許改動檔案清單」不得重疊，重疊就改串行。
- 平行實作 agent 上限 3 個（驗收與統整成本會超過收益）；scout／verifier 等唯讀 agent 不設限。

## §6 失敗軌跡的保存

升級時必附：每次嘗試的（a）派工單原文（b）agent 回報原文（c）實際錯誤輸出。禁止只寫「前面失敗了」一句話——上層模型沒有失敗軌跡就會重走同一條死路。子任務結束後，若教訓可重複利用 → 依 F 協議寫入 LESSONS。
