# F — 知識迭代與反思協議

> FROZEN。定義：誰能改哪些檔、踩坑怎麼記、知識堆積到什麼門檻怎麼精簡。檔案清單與權限以 INDEX 的表為準，本檔定義操作規則。

## §1 修改權限速查

| 動作 | 准許 |
|---|---|
| LESSONS.md 追加一條踩坑（依 §2 格式） | ✅ 自行做，不必問 |
| PLAYBOOK.md 追加一條模式（依該檔頭格式，四欄齊全） | ✅ 自行做；缺欄不得收錄 |
| PROPOSALS.md 追加提案 | ✅ 自行做（提案本身不需同意，執行才需要） |
| SPEC.md／OPS.md 在功能或部署變動後同步更新 | ✅ 自行做，commit 訊息註明對應改動 |
| ENV-FACTS.md 更正事實 | ✅ 但必附新證據＋日期；只准改內容，不准動 § 編號 |
| CLAUDE.md 路由表**追加一列**（新檔案落地時） | ✅ 但先過 §5 自檢三問，且總行數不得破 120 |
| 改 CLAUDE.md 其他部分、任何 docs/harness/ 檔、harness-config.json、MIGRATION-LOG | ❌ 寫 PROPOSALS 等 User 裁決 |
| 改 .claude/settings*.json、啟用 hooks、全域設定 | ❌ 同上 |

## §2 踩坑紀錄（LESSONS）固定格式

**何時必須寫**：一個問題耗掉 ≥2 輪派工才解決、或「未來別的 session 很可能再踩」。純手滑打錯字不用記。

```markdown
## L-【下一個流水號】 【一句話標題】（YYYY-MM-DD）
- 症狀：【User 或 agent 看到的表面現象，含錯誤訊息原文】
- 根因：【真正原因，不是「代碼有 bug」這種空話】
- 解法：【實際做了什麼】
- 防再犯：【結構性預防：規則？腳本？檢查？——沒有就寫「無，只能靠本條記錄」】
- 證據/出處：【檔案:行號 或 指令輸出 或 對話日期】
```

編號遞增不重用；寫完不需要通知 User（/reflect 摘要時會帶到）。

## §3 LESSONS vs PLAYBOOK 怎麼分

- **LESSONS**＝「別再踩」：一次性的坑、環境陷阱、錯誤方向。
- **PLAYBOOK**＝「照著做」：已驗證、會重複使用的解法配方，四欄齊全（適用情境／步驟／驗證方式／不適用情況）才收。
- 同一事件可以兩邊各寫一條（坑記 LESSONS，解法配方記 PLAYBOOK），互相引用編號。

## §4 精簡與抽象化門檻（防知識檔腐化）

| 觸發 | 動作 |
|---|---|
| LESSONS 超過 30 條或 300 行 | 產出精簡草案：合併同類條目、把 ≥3 條同型教訓抽象成一條原則；草案寫進 PROPOSALS 等 User 同意後替換（原檔先 .bak） |
| PLAYBOOK 某條目連續 3 次套用失敗 | 標記 `[STALE]` ＋寫一條 LESSONS 記錄失效原因；不删除，等 User 裁決 |
| ENV-FACTS 某條事實被實測推翻 | 立即更正（附證據＋日期），並檢查引用它的 LESSONS/PLAYBOOK 是否連帶失效 |
| PROPOSALS 累積 ≥5 條未裁決 | /reflect 時主動向 User 彙報清單，請求批次裁決 |

## §5 CLAUDE.md 追加自檢三問（全部答完才准動）

1. 這條能不能**併入既有條目**？（能→併，不加新列）
2. 這條是不是其實該放進 SPEC／OPS／LESSONS，路由表已經到得了？（是→放那裡，不動 CLAUDE.md）
3. 加了這行，**要刪哪一行**來維持 ≤120？（答不出來→寫 PROPOSALS）

## §6 /reflect 收工清單（每個工作 session 結束時跑）

1. 本次有沒有 §2 定義的「必須記」事件？→ 寫 LESSONS。
2. 有沒有高階模型解出的可重複模式？→ 寫 PLAYBOOK（四欄齊才收）。
3. SPEC／OPS 有沒有因本次改動而過時的段落？→ 同步。
4. 有沒有想改而不能改的規則？→ 寫 PROPOSALS。
5. 向 User 一段話總結：做了什麼、記了什麼、有哪些提案待裁決。
