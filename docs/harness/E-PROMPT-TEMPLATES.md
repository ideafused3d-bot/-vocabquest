# E — 標準化派工 Prompt 模板（四式）

> FROZEN。派工時**複製對應模板填空**，不得徒手寫派工 prompt（C 守則 §2）。【】為必填空格；「先讀」欄只給路徑不貼全文。
> 所有模板共用的紅線：專案事實查 ENV-FACTS §N；危險操作見 ENV-FACTS §4；不確定標 `UNVERIFIED`；回報禁貼 >15 行代碼。

## 模板 1：搜尋研究（派給 scout／Explore，唯讀）

```text
【任務】回答這個問題：【要查明的具體問題，一句話】
【背景】因為【為什麼需要知道】。
【先讀】從這些地方開始（不限於此）：【路徑1】【路徑2】；專案目錄地圖見 docs/project/ENV-FACTS.md §6。
【禁止】任何寫入或修改；把整檔內容貼回來。
【回報格式】≤40 行：
1. 直接答案（1–3 句）
2. 證據清單：每條「檔案:行號 — 一句話說明」
3. 未能確認的部分，逐條標 UNVERIFIED
```

## 模板 2：功能實作（派給 builder）

```text
【任務】【一句話目標】
【背景】【為什麼做＋相關規格在哪：docs/project/SPEC.md 的哪一節】
【先讀】docs/project/ENV-FACTS.md §3（驗證指令）、§4（危險操作）＋【本任務相關檔案路徑】
【允許改動的檔案】【明確清單；改這清單以外的檔案須在回報中逐檔說明理由】
【禁止】碰 ENV-FACTS §4 所列項目；修改驗收條件本身；順手重構無關代碼。
【驗收條件】（可判定真假，逐條列）
1. 【功能行為：給定 X 輸入，產生 Y 結果】
2. ENV-FACTS §3 最低驗證組合全過（exit 0）
3. 【若適用：依 docs/project/PLAYBOOK.md PB-XXX 的驗證方式跑過】
【回報格式】
- STATUS: READY_FOR_VERIFY（不得自稱「完成」，見 D 矩陣 §2.7）
- 改動清單：每項「檔案:行號範圍 — 一句話」
- 驗證指令輸出末 5 行
- 逐條驗收條件：過／不過＋證據
- 標 UNVERIFIED 的事項
```

## 模板 3：代碼重構（派給 builder；行為不得改變）

```text
【任務】重構【目標範圍】，行為零改變。
【背景】【重構動機：重複／耦合／效能，附證據 檔案:行號】
【先讀】docs/project/ENV-FACTS.md §3、【相關檔案路徑】
【允許改動的檔案】【清單】
【行為凍結證明】重構前先記錄基準：【怎麼取得目前行為的快照，例：對純函式跑 PLAYBOOK PB-XXX 式斷言腳本並存輸出】；重構後同一腳本輸出必須逐字元一致。
【禁止】改公開介面／函式簽名（除非驗收條件明列）；改動行為「順便修 bug」——發現 bug 記下來回報，不要修。
【驗收條件】
1. 行為凍結證明：前後輸出一致（貼 diff 為空的證據）
2. ENV-FACTS §3 最低驗證組合全過
3. 【量化改善指標，例：重複代碼區塊從 N 處降到 1 處】
【回報格式】同模板 2。
```

## 模板 4：代碼審查（派給 verifier／reviewer，唯讀）

```text
【任務】審查【分支或改動範圍】，回報缺陷，不要動手修。
【背景】這批改動的目標是【原任務一句話】；驗收條件如下：【貼上原驗收條件】
【先讀】git diff 範圍：【指令或檔案清單】＋ docs/project/SPEC.md 相關節＋ docs/project/LESSONS.md（查是否踩過同款坑）
【審查清單】（逐項回答）
1. read-back：成果檔案真實存在、非截斷（檔尾正常收束）？
2. 實跑：ENV-FACTS §3 驗證指令實際執行結果？（貼 exit code＋末 5 行）
3. 驗收條件逐條：過／不過＋證據
4. 改動範圍：git status 是否只含允許清單內的檔案？
5. 有無違反 CLAUDE.md 紅線或 ENV-FACTS §4？
6. 有無編造信號（引用不存在的測試／函式／路徑）？
【回報格式】≤40 行：VERDICT: PASS 或 FAIL（FAIL 列出最小修復清單，每條「檔案:行號 — 問題一句話」）。禁止給「建議性美化意見」混進 FAIL 清單——缺陷與品味分開列。
```

## 使用備註

- 模板 1/4 是唯讀任務，可平行多開；模板 2/3 受 C 守則 §5 平行上限約束。
- 派工前自檢：三件套齊了嗎（目標背景／驗收條件／回報格式）？驗收條件每一條都能判定真假嗎？
- agent 回報不合回報格式（貼大段代碼、無證據）→ 視同該次失敗，計入 C 守則 §3 的失敗次數。
