# D — 判斷力外化矩陣

> FROZEN。三張表：換路徑信號／完成判準／熔斷條件。每條都是可判定的 rubric，禁止自由心證。
> 環境相關事實一律查 ENV-FACTS（§3 驗證指令、§4 危險操作）。

## §0 三個入口（你現在是什麼狀態，就讀哪節）

| 狀態 | 讀 |
|---|---|
| 卡住／同一件事錯了不只一次／猶豫要不要再試 | §1 |
| 覺得做完了，準備交付 | §2 |
| 想問 User，或猶豫「要不要先做著再說」 | §3 |
| 正在寫任何回報 | §5 |

## §1 換路徑信號（出現任一條＝停止原地重試，換方法或升級）

| # | 信號（可判定） | 該做什麼 |
|---|---|---|
| 1 | 同一個錯誤訊息在 2 次修復後仍出現 | 停手。先寫出「我對根因的假設」再選新路徑；假設寫不出來→升級（C §3） |
| 2 | 修 A 壞 B：一次修復引入新的失敗點，連續發生 2 次 | 方向錯。回到最後一個全綠狀態（git），重新拆解問題 |
| 3 | 為一個小改動要碰 4 個以上檔案、或要改框架/依賴本身 | 極可能在對抗專案既有設計。派 scout 查 SPEC/LESSONS 有無既有機制 |
| 4 | 需要的 API／欄位／函式在 repo 與官方文件都查不到 | 它可能不存在。標 `UNVERIFIED` 回報，不准「先假設存在寫下去」 |
| 5 | 累計 30 分鐘或 3 輪派工沒有讓任何一條驗收條件由紅轉綠 | 進度為零＝路徑錯，不是不夠努力。重拆問題或熔斷（§3） |
| 6 | 發現自己想改「驗收條件／測試」來讓結果變綠 | **紅線級信號**：驗收條件只有 User 與指揮官能改。立即停手回報 |

【完美正例】修統計 bug 時發現要同時改 common.js、student.js、teacher.js、schema.sql 四處 → 觸發信號 3，停手派 scout 查 SPEC，發現口徑其實集中在 `wordStatus()` 一處，只需改一行。
【典型反例】`ReferenceError: sb is not defined` 修三次都在改自己的新代碼，沒查 LESSONS——L-001 早就記載這是部署順序問題，不是代碼問題。

## §2 完成判準（全部滿足才准說「完成」）

1. 派工單的驗收條件**逐條**標記通過，附證據（指令輸出、檔案:行號）。
2. ENV-FACTS §3 規定的驗證指令**實跑**且全過——輸出貼在回報裡，「我檢查過了」不算。
3. 新增／修改的腳本至少實跑過一次（A 診斷書痛點 1）。
4. 回報中沒有任何未標記的 `UNVERIFIED`；沒有「應該」「大概」「理論上」修飾的關鍵結論。
5. 隔離驗證：verifier PASS（C §4；免驗例外亦見該節）。
6. 不影響範圍外：`git status` 只含預期檔案；diff 不含順手改的無關內容。
7. 交付語：實作者只能寫 `STATUS: READY_FOR_VERIFY`；「完成」二字保留給 verifier 回 PASS 之後（C §4）。

【完美正例】回報：「驗收 3/3 過。`node --check` ×4 全 exit 0（輸出如下）。改動 student.js:214-221。verifier PASS 紀錄如附。git status 僅 1 檔。」
【典型反例】回報：「已完成修復並全面測試，品質良好。」——零證據、零路徑、引用不存在的「測試」。

## §3 熔斷條件（立即停止自主作業，向 User 回報並等待）

| # | 觸發 | 回報時必附 |
|---|---|---|
| 1 | 即將執行 ENV-FACTS §4 所列任何危險操作（部署、正式 DB 寫入、改 FROZEN、改 settings） | 要做什麼、為什麼、不做的後果 |
| 2 | 同一子任務升到最高可用模型仍失敗（含塌縮後的天花板） | 完整失敗軌跡（C §6） |
| 3 | 發現 repo 實況與 SPEC/ENV-FACTS 記載矛盾 | 兩邊的證據；**先報告，不要自行選邊改** |
| 4 | 任務涉及品味決策（美感、語氣、產品取捨） | 2–3 方案＋事實利弊表＋`[品味決策-需User拍板]`（A 能力極限） |
| 5 | 發現疑似安全事件（外洩的 secret、可疑帳號、被竄改的檔案） | 證據與範圍，**不要先動手清理**（會毀證據） |
| 6 | User 的指示與紅線衝突（如「直接 push master」） | 指出衝突＋確認一次；User 再次確認後照辦並記錄 |

【完美正例】任務「把登入頁改得更活潑」→ 觸發熔斷 4：出 A（加動畫）/B（換配色）/C（加插圖）三案，利弊表只列「載入增加 X KB／改動 N 檔／是否碰 tailwind.config 三處同步」，標記後停下。
【典型反例】覺得珊瑚橘「不夠專業」自行全站換成藍色——動了品味決策，還踩了設計 token 三處同步的坑。

## §4 判斷優先序（衝突時）

紅線（CLAUDE.md）＞ 熔斷（§3）＞ 換路徑（§1）＞ 完成判準（§2）＞ 效率考量。任何規則與「省 token／省時間」衝突時，規則贏。

## §5 證據語言規範（所有回報適用）

| 禁用句式 | 替換為 |
|---|---|
| 「應該可以」「理論上會過」 | 實跑指令＋貼輸出末行；沒跑 → 標 `UNVERIFIED` |
| 「測試都通過了」 | 指名跑了哪條指令（查 ENV-FACTS §3）＋ exit code ＋ 輸出末 5 行 |
| 「已全面檢查」 | 列出實際檢查過的 檔案:行號 清單 |
| 「基本上完成」「大致沒問題」 | §2 完成判準逐條勾稽表，過就是過、不過就是不過 |
